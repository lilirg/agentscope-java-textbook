## 11.5.1 Docker容器化部署相关陷阱

### 11.5.1.1 陷阱1：Docker镜像过大

**现象**：
```
Warning: Docker image size is too large (>500MB)
```

**原因**：
- 依赖库过多
- 未使用多阶段构建
- 包含调试文件

**调试方法**：
```bash
# 1. 检查镜像大小
docker images

# 2. 检查镜像层
docker history agent-app:latest
```

**解决方案**：

| 方案 | 说明 | 适用场景 |
|------|------|---------|
| **多阶段构建** | 使用多阶段构建减小镜像体积 | 通用 |
| **精简基础镜像** | 使用`alpine`镜像 | 生产环境 |
| **排除调试文件** | 移除调试依赖 | 生产环境 |

**示例**：
```dockerfile
# Dockerfile（多阶段构建）
# 第一阶段：构建
FROM maven:3.8-openjdk-17 AS builder
WORKDIR /build
COPY pom.xml .
RUN mvn dependency:go-offline -B
COPY src ./src
RUN mvn package -DskipTests

# 第二阶段：运行
FROM openjdk:17-jdk-slim
WORKDIR /app
COPY --from=builder /build/target/app.jar app.jar
EXPOSE 8080
ENTRYPOINT ["java", "-jar", "app.jar"]
```

### 11.5.1.2 陷阱2：容器无法启动

**现象**：
```
Error: Container exited with code 1
```

**原因**：
- JVM参数配置错误
- 端口冲突
- 环境变量缺失

**调试方法**：
```bash
# 1. 查看容器日志
docker logs <container-id>

# 2. 检查容器状态
docker ps -a

# 3. 进入容器调试
docker exec -it <container-id> /bin/bash
```

**解决方案**：

| 方案 | 说明 | 适用场景 |
|------|------|---------|
| **检查JVM参数** | 确保JVM参数正确 | 内存配置错误 |
| **检查端口** | 确保端口未被占用 | 端口冲突 |
| **检查环境变量** | 确保环境变量配置 | 环境变量缺失 |

**示例**：
```dockerfile
# Dockerfile
FROM openjdk:17-jdk-slim

WORKDIR /app

COPY target/app.jar app.jar

# ✅ 配置JVM参数
ENV JAVA_OPTS="-Xms256m -Xmx512m -XX:+UseG1GC"

EXPOSE 8080

# ✅ 使用JAVA_OPTS
ENTRYPOINT ["sh", "-c", "java $JAVA_OPTS -jar app.jar"]
```

## 11.5.2 Prometheus监控系统相关陷阱

### 11.5.2.1 陷阱3：Prometheus数据采集失败

**现象**：
```
Warning: Prometheus failed to scrape metrics
```

**原因**：
- Actuator端点未暴露
- Prometheus配置错误
- 网络问题

**调试方法**：
```bash
# 1. 检查Actuator端点
curl http://localhost:8080/actuator/metrics

# 2. 检查Prometheus配置
docker exec -it prometheus cat /etc/prometheus/prometheus.yml

# 3. 检查网络连接
docker exec -it prometheus curl http://agent-app:8080/actuator/metrics
```

**解决方案**：

| 方案 | 说明 | 适用场景 |
|------|------|---------|
| **暴露Actuator端点** | 确保`management.endpoints.web.exposure.include=health,info,metrics` | Actuator未暴露 |
| **检查Prometheus配置** | 确保`targets`配置正确 | Prometheus配置错误 |
| **检查网络** | 确保Prometheus能访问Agent应用 | 网络问题 |

**示例**：
```yaml
# application.yml
management:
  endpoints:
    web:
      exposure:
        include: health,info,metrics  # ✅ 暴露所有端点
  metrics:
    tags:
      application: ${spring.application.name}
```

| **检查认证** | 确保认证信息正确 | 认证问题 |

**示例**：

## 11.5.3 告警系统相关陷阱

### 11.5.3.1 陷阱5：告警未触发

**现象**：
```
Warning: Alert not triggered despite exceeding threshold
```

**原因**：
- 告警规则配置错误
- AlertManager配置错误
- 钉钉/邮件配置错误

**调试方法**：
```bash
# 1. 检查Prometheus告警规则
curl http://prometheus:9090/api/v1/rules

# 2. 检查AlertManager配置
docker exec -it alertmanager cat /etc/alertmanager/alertmanager.yml

# 3. 测试钉钉/邮件通知
curl -X POST http://localhost:8080/test-alert
```

**解决方案**：

| 方案 | 说明 | 适用场景 |
|------|------|---------|
| **检查告警规则** | 确保Prometheus告警规则正确 | 告警规则错误 |
| **检查AlertManager** | 确保AlertManager配置正确 | AlertManager配置错误 |
| **检查通知配置** | 确保钉钉/邮件配置正确 | 通知配置错误 |

**示例**：
```yaml
# prometheus.yml
alerting:
  alertmanagers:
    - static_configs:
        - targets: ['alertmanager:9093']
```

```yaml
# alertmanager.yml
route:
  receiver: 'dingtalk'
  routes:
    - match:
        severity: warning
      receiver: 'dingtalk'
receivers:
  - name: 'dingtalk'
    dingtalk_configs:
      - webhook: 'https://oapi.dingtalk.com/robot/send?access_token=xxx'
```

## 11.5.4 日志分析相关陷阱

### 11.5.4.1 陷阱6：ELK日志索引失败

**现象**：
```
Error: Failed to index logs to Elasticsearch
```

**原因**：
- Logstash配置错误
- Elasticsearch配置错误
- 日志格式错误

**调试方法**：
```bash
# 1. 检查Logstash日志
docker logs logstash

# 2. 检查Elasticsearch日志
docker logs elasticsearch

# 3. 测试Elasticsearch连接
curl -X GET http://elasticsearch:9200/_cluster/health
```

**解决方案**：

| 方案 | 说明 | 适用场景 |
|------|------|---------|
| **检查Logstash配置** | 确保Logstash配置正确 | Logstash配置错误 |
| **检查Elasticsearch配置** | 确保Elasticsearch配置正确 | Elasticsearch配置错误 |
| **检查日志格式** | 确保日志格式正确 | 日志格式错误 |

**示例**：
```ruby
# logstash.conf
filter {
  grok {
    match => { "message" => "%{TIMESTAMP_ISO8601:timestamp} %{LOGLEVEL:level} %{JAVACLASS:class} - %{GREEDYDATA:message}" }
  }
  date {
    match => [ "timestamp", "yyyy-MM-dd HH:mm:ss" ]
    target => "@timestamp"
  }
}
```

## 11.5.5 Kubernetes部署相关陷阱

### 11.5.5.1 陷阱7：K8s Pod无法启动

**现象**：
```
Error: Pod is in CrashLoopBackOff state
```

**原因**：
- 镜像拉取失败
- 健康检查失败
- 资源不足

**调试方法**：
```bash
# 1. 查看Pod状态
kubectl get pods

# 2. 查看Pod日志
kubectl logs <pod-name>

# 3. 查看Pod事件
kubectl describe pod <pod-name>
```

**解决方案**：

| 方案 | 说明 | 适用场景 |
|------|------|---------|
| **检查镜像** | 确保镜像地址正确 | 镜像拉取失败 |
| **检查健康检查** | 确保健康检查配置正确 | 健康检查失败 |
| **检查资源** | 确保资源请求配置正确 | 资源不足 |

**示例**：
```yaml
# deployment.yaml
livenessProbe:
  httpGet:
    path: /actuator/health
    port: 8080
  initialDelaySeconds: 10
  periodSeconds: 5
readinessProbe:
  httpGet:
    path: /actuator/ready
    port: 8080
  initialDelaySeconds: 5
  periodSeconds: 3
```

## 11.5.6 通用调试技巧

### 11.5.6.1 单元测试（JUnit + Mockito）

**测试示例**（告警服务）：
```java
import org.junit.jupiter.api.Test;
import org.mockito.Mockito;

import static org.mockito.Mockito.*;

public class AlertServiceTest {
    
    @Test
    public void testNotify() {
        // 1. 创建Mock
        DingTalkNotifier dingTalkNotifier = Mockito.mock(DingTalkNotifier.class);
        EmailNotifier emailNotifier = Mockito.mock(EmailNotifier.class);
        
        AlertService alertService = new AlertService(dingTalkNotifier, emailNotifier);
        
        // 2. 执行
        alertService.notify("Test Alert", "Test Message");
        
        // 3. 验证
        verify(dingTalkNotifier, times(1)).notify("Test Alert", "Test Message");
    }
}
```
