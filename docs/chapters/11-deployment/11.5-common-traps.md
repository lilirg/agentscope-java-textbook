## 部署陷阱

### 陷阱1：Docker容器崩溃

**问题描述**：
```
问题：AI应用Docker容器频繁崩溃
症状：
- 容器重启频繁
- 应用无法启动
- 日志缺失

原因：
- 内存不足
- 端口冲突
- 环境变量未设置
```

**解决方案**：

| 方案 | 说明 | 适用场景 |
|------|------|---------|
| **资源限制** | 设置容器内存限制 | 资源受限场景 |
| **端口检查** | 检查端口冲突 | 多服务部署场景 |
| **环境变量** | 使用.env文件设置环境变量 | 通用场景 |

**示例代码**：
```java
// Docker资源限制
# docker-compose.yml
version: '3.8'
services:
  ai-app:
    image: ai-app:latest
    mem_limit: 4g
    memswap_limit: 4g
    deploy:
      resources:
        reservations:
          memory: 2g
        limits:
          memory: 4g

# Dockerfile
FROM openjdk:17-jdk-slim
WORKDIR /app
COPY target/app.jar app.jar
EXPOSE 8080
ENTRYPOINT ["java", "-Xms2g", "-Xmx4g", "-jar", "app.jar"]
```

### 陷阱2：CI/CD流水线失败

**问题描述**：
```
问题：CI/CD流水线构建/部署失败
症状：
- 构建失败
- 部署失败
- 回滚失败

原因：
- 构建脚本错误
- 依赖未下载
- 环境配置错误
```

**解决方案**：

| 方案 | 说明 | 适用场景 |
|------|------|---------|
| **构建脚本验证** | 验证构建脚本正确性 | 开发/测试环境 |
| **依赖缓存** | 使用依赖缓存加速构建 | 高频构建场景 |
| **环境隔离** | 使用环境隔离避免冲突 | 多环境部署场景 |

**示例代码**：
```yaml
# .github/workflows/deploy.yml
name: Deploy AI App
on:
  push:
    branches: [main]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      
      - name: Setup JDK 17
        uses: actions/setup-java@v3
        with:
          java-version: '17'
          cache: maven
      
      - name: Build with Maven
        run: mvn clean package -DskipTests
      
      - name: Build Docker image
        run: docker build -t ai-app:${{ github.sha }} .
      
      - name: Push Docker image
        run: |
          docker login -u ${{ secrets.DOCKER_USERNAME }} -p ${{ secrets.DOCKER_PASSWORD }}
          docker push ai-app:${{ github.sha }}
  
  deploy:
    needs: build
    runs-on: ubuntu-latest
    steps:
      - name: Deploy to Kubernetes
        run: |
          kubectl set image deployment/ai-app ai-app=ai-app:${{ github.sha }}
```

## 监控陷阱

### 陷阱3：监控指标缺失

**问题描述**：
```
问题：监控缺乏关键指标，无法及时发现故障
症状：
- 故障发现滞后
- 无法定位问题
- 缺乏监控基线

原因：
- 未监控关键指标
- 未设置告警阈值
- 未记录历史数据
```

**解决方案**：

| 方案 | 说明 | 适用场景 |
|------|------|---------|
| **关键指标监控** | 监控CPU、内存、请求量、错误率 | 通用场景 |
| **告警阈值设置** | 设置告警阈值（如CPU>80%） | 生产环境 |
| **历史数据分析** | 记录并分析历史监控数据 | 性能调优场景 |

**示例代码**：
```java
// 关键指标监控
public class SystemMonitor {
    private finalOperatingSystemMXBean osBean = ManagementFactory.getOperatingSystemMXBean();
    private final RuntimeMXBean runtimeBean = ManagementFactory.getRuntimeMXBean();
    
    public Map<String, Object> getSystemMetrics() {
        Map<String, Object> metrics = new HashMap<>();
        
        // CPU使用率
        metrics.put("cpu_usage", osBean.getSystemCpuLoad() * 100);
        
        // 内存使用率
        Runtime runtime = Runtime.getRuntime();
        long totalMemory = runtime.totalMemory();
        long maxMemory = runtime.maxMemory();
        long freeMemory = runtime.freeMemory();
        
        metrics.put("memory_used", totalMemory - freeMemory);
        metrics.put("memory_max", maxMemory);
        metrics.put("memory_usage", (double)(totalMemory - freeMemory) / maxMemory * 100);
        
        // JVM启动时间
        metrics.put("jvm_uptime_ms", runtimeBean.getUptime());
        
        return metrics;
    }
}

// 告警阈值设置
public class AlertingService {
    private static final double CPU_THRESHOLD = 80.0;
    private static final double MEMORY_THRESHOLD = 80.0;
    private static final int REQUEST_RATE_THRESHOLD = 1000; // req/s
    
    private final SystemMonitor monitor;
    
    public AlertingService(SystemMonitor monitor) {
        this.monitor = monitor;
    }
    
    public void checkAlerts() {
        Map<String, Object> metrics = monitor.getSystemMetrics();
        
        // CPU告警
        double cpuUsage = (double) metrics.get("cpu_usage");
        if (cpuUsage > CPU_THRESHOLD) {
            logger.warn("CPU usage too high: {}% > {}%", cpuUsage, CPU_THRESHOLD);
        }
        
        // 内存告警
        double memoryUsage = (double) metrics.get("memory_usage");
        if (memoryUsage > MEMORY_THRESHOLD) {
            logger.warn("Memory usage too high: {}% > {}%", memoryUsage, MEMORY_THRESHOLD);
        }
    }
}

// 历史数据分析
public class HistoricalDataAnalyzer {
    private final List<Map<String, Object>> history = new ArrayList<>();
    
    public void recordMetrics(Map<String, Object> metrics) {
        history.add(metrics);
        
        // 限制历史数据量
        if (history.size() > 10000) {
            history.subList(0, 1000).clear();
        }
    }
    
    public Map<String, Double> getAverageMetrics() {
        if (history.isEmpty()) {
            return Collections.emptyMap();
        }
        
        Map<String, Double> averages = new HashMap<>();
        
        // 计算每个指标的平均值
        for (String key : history.get(0).keySet()) {
            double sum = history.stream()
                .map(m -> (Double) m.get(key))
                .mapToDouble(Double::doubleValue)
                .sum();
            
            averages.put(key, sum / history.size());
        }
        
        return averages;
    }
}
```

### 陷阱4：日志分析困难

**问题描述**：
```
问题：日志量大、格式混乱，难以分析
症状：
- 日志难以搜索
- 日志解析困难
- 关键信息丢失

原因：
- 日志格式不统一
- 日志量过大
- 未使用日志集中管理系统
```

**解决方案**：

| 方案 | 说明 | 适用场景 |
|------|------|---------|
| **日志格式统一** | 使用JSON格式记录日志 | 通用场景 |
| **日志采样** | 对高频日志进行采样 | 高日志量场景 |
| **集中管理** | 使用ELK/Graylog集中管理 | 生产环境 |

**示例代码**：
```java
// JSON日志格式
import com.fasterxml.jackson.databind.ObjectMapper;
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;

public class JsonLoggingService {
    private static final Logger logger = LoggerFactory.getLogger(JsonLoggingService.class);
    private static final ObjectMapper objectMapper = new ObjectMapper();
    
    public void logRequest(String requestId, String endpoint, long duration) {
        Map<String, Object> log = new HashMap<>();
        log.put("timestamp", Instant.now().toString());
        log.put("request_id", requestId);
        log.put("endpoint", endpoint);
        log.put("duration_ms", duration);
        
        try {
            logger.info("Request log: {}", objectMapper.writeValueAsString(log));
        } catch (JsonProcessingException e) {
            logger.error("Failed to serialize log", e);
        }
    }
}

// 日志采样
public class SamplingLogger {
    private static final Logger logger = LoggerFactory.getLogger(SamplingLogger.class);
    private static final int SAMPLING_RATE = 10; // 每10条日志记录1条
    
    private int counter = 0;
    
    public void log(String message) {
        if (counter++ % SAMPLING_RATE == 0) {
            logger.info(message);
        }
    }
}

// 集中管理（示例配置）
# logback-spring.xml
<configuration>
    <appender name="JSON_FILE" class="ch.qos.logback.core.rolling.RollingFileAppender">
        <file>logs/app.json</file>
        <encoder class="net.logstash.logback.encoder.LoggingEventCompositeJsonEncoder">
            <providers>
                <timestamp/>
                <version/>
                <loggerName/>
                <message/>
                <logLevel/>
                <threadName/>
                <context/>
                <stackTrace/>
            </providers>
        </encoder>
    </appender>
    
    <root level="INFO">
        <appender-ref ref="JSON_FILE"/>
    </root>
</configuration>
```

## 告警陷阱

### 陷阱5：告警风暴

**问题描述**：
```
问题：告警过多，导致告警风暴
症状：
- 告警通知过多
- 无法区分重要告警
- 告警疲劳

原因：
- 告警阈值过低
- 未设置告警抑制
- 未分级告警
```

**解决方案**：

| 方案 | 说明 | 适用场景 |
|------|------|---------|
| **告警分级** | 设置告警级别（如P0、P1、P2） | 通用场景 |
| **告警抑制** | 设置告警抑制时间（如5分钟内不重复告警） | 高频告警场景 |
| **告警聚合** | 聚合相同类型告警 | 多系统告警场景 |

**示例代码**：
```java
// 告警分级
public enum AlertLevel {
    P0("Critical", 1),
    P1("High", 2),
    P2("Medium", 3),
    P3("Low", 4);
    
    private final String description;
    private final int priority;
    
    AlertLevel(String description, int priority) {
        this.description = description;
        this.priority = priority;
    }
    
    public String getDescription() {
        return description;
    }
    
    public int getPriority() {
        return priority;
    }
}

// 告警抑制
public class AlertSuppression {
    private final Map<String, Long> lastAlertTime = new ConcurrentHashMap<>();
    private static final long SUPPRESSION_INTERVAL = 300000; // 5分钟
    
    public boolean shouldAlert(String alertKey) {
        Long lastTime = lastAlertTime.get(alertKey);
        if (lastTime == null) {
            lastAlertTime.put(alertKey, System.currentTimeMillis());
            return true;
        }
        
        if (System.currentTimeMillis() - lastTime > SUPPRESSION_INTERVAL) {
            lastAlertTime.put(alertKey, System.currentTimeMillis());
            return true;
        }
        
        return false;
    }
}

// 告警聚合
public class AlertAggregation {
    private final Map<String, Integer> alertCounts = new ConcurrentHashMap<>();
    private final ScheduledExecutorService scheduler = Executors.newSingleThreadScheduledExecutor();
    
    public AlertAggregation() {
        // 每5分钟聚合告警
        scheduler.scheduleAtFixedRate(this::aggregateAlerts, 0, 5, TimeUnit.MINUTES);
    }
    
    public void recordAlert(String alertKey) {
        alertCounts.compute(alertKey, (k, v) -> v != null ? v + 1 : 1);
    }
    
    private void aggregateAlerts() {
        alertCounts.forEach((key, count) -> {
            if (count > 1) {
                logger.warn("Alert {} occurred {} times in the last 5 minutes", key, count);
            } else {
                logger.warn("Alert {} occurred", key);
            }
        });
        
        alertCounts.clear();
    }
}
```

## 通用调试技巧

### 陷阱6：部署调试工具不足

**问题描述**：
```
问题：部署调试缺乏足够工具，无法快速定位问题
症状：
- 难以定位部署问题
- 无法查看容器日志
- 无法连接容器进行调试
```

**解决方案**：

| 方案 | 说明 | 适用场景 |
|------|------|---------|
| **容器日志查看** | 使用docker logs查看日志 | 开发/测试环境 |
| **容器调试** | 使用kubectl exec连接容器 | 生产环境 |
| **健康检查** | 设置容器健康检查 | 所有环境 |

**示例代码**：
```yaml
# Docker健康检查
FROM openjdk:17-jdk-slim
WORKDIR /app
COPY target/app.jar app.jar
EXPOSE 8080

HEALTHCHECK --interval=30s --timeout=3s --start-period=5s --retries=3 \
  CMD curl -f http://localhost:8080/actuator/health || exit 1

ENTRYPOINT ["java", "-jar", "app.jar"]
```

```yaml
# Kubernetes健康检查
apiVersion: v1
kind: Pod
metadata:
  name: ai-app
spec:
  containers:
  - name: ai-app
    image: ai-app:latest
    ports:
    - containerPort: 8080
    livenessProbe:
      httpGet:
        path: /actuator/health
        port: 8080
      initialDelaySeconds: 10
      periodSeconds: 10
    readinessProbe:
      httpGet:
        path: /actuator/health
        port: 8080
      initialDelaySeconds: 5
      periodSeconds: 5
```
