## 短期记忆陷阱

### 陷阱1：短期记忆窗口溢出

**问题描述**：
```
问题：短期记忆窗口大小固定为20条，但token总数可能超出模型限制
症状：
- API调用失败（prompt too long）
- LLM响应截断
- 对话历史丢失

原因：
- 每条消息平均长度未知
- 未实时计算token数量
- 固定窗口策略未考虑消息长度
```

**解决方案**：

| 方案 | 说明 | 适用场景 |
|------|------|---------|
| **摘要压缩** | 使用LLM生成摘要，减少token消耗 | 长对话、重要消息保留 |
| **降低窗口大小** | 调整`maxMessages`参数（如10） | 短对话、快速响应 |
| **截断正文** | 保留消息头，截断正文 | 冗长消息、非关键信息 |

**示例代码**：
```java
// 计算token数量（简化版）
public int countTokens(List<Message> messages) {
    int totalTokens = 0;
    for (Message msg : messages) {
        // 简单估算：1个汉字 ≈ 1.3 tokens
        totalTokens += (int) (msg.getContent().length() * 1.3);
    }
    return totalTokens;
}

// 摘要压缩策略
public void compressMemory() {
    if (countTokens(history) > MAX_TOKENS) {
        // 生成摘要
        String summary = llm.generate("将以下对话压缩为3句话摘要：\n" + 
            history.stream().map(m -> m.role + ": " + m.content).collect(Collectors.joining("\n")));
        
        // 替换为摘要
        history.clear();
        history.add(new Message("system", "[摘要] " + summary));
    }
}
```

### 陷阱2：线程安全问题

**问题描述**：
```
问题：ShortTermMemory未考虑并发访问，可能导致数据不一致
症状：
- 消息丢失
- 消息重复
- 窗口大小异常

原因：
- LinkedList非线程安全
- 多线程同时add/getRecent()
```

**解决方案**：

| 方案 | 说明 | 适用场景 |
|------|------|---------|
| **synchronized** | 同步方法，确保线程安全 | 低并发场景 |
| **CopyOnWriteArrayList** | 写时复制，读写分离 | 高读低写场景 |
| **ConcurrentLinkedQueue** | 并发队列，高性能 | 高并发场景 |

**示例代码**：
```java
// 方案1：synchronized同步
public synchronized void add(Message msg) {
    history.addLast(msg);
    while (history.size() > maxMessages) {
        history.removeFirst();
    }
}

public synchronized List<Message> getRecent() {
    return new ArrayList<>(history);
}

// 方案2：CopyOnWriteArrayList
private final CopyOnWriteArrayList<Message> history = new CopyOnWriteArrayList<>();

public void add(Message msg) {
    history.add(msg);
    while (history.size() > maxMessages) {
        history.remove(0);
    }
}

public List<Message> getRecent() {
    return new ArrayList<>(history);
}
```

## 长期记忆陷阱

### 陷阱3：JSON序列化失败

**问题描述**：
```
问题：LongTermMemory序列化/反序列化失败
症状：
- Jackson抛出`InvalidDefinitionException`
- JSON格式错误
- 数据丢失

原因：
- 字段类型不匹配（如Map<String, Object>）
- 循环引用（self-reference）
- 未配置ObjectMapper
```

**解决方案**：

| 方案 | 说明 | 适用场景 |
|------|------|---------|
| **配置ObjectMapper** | 忽略循环引用、自定义序列化规则 | 通用场景 |
| **简化数据结构** | 使用POJO替代Map<String, Object> | 结构固定场景 |
| **自定义Serializer** | 实现JsonSerializer接口 | 复杂类型场景 |

**示例代码**：
```java
// 配置ObjectMapper
ObjectMapper objectMapper = new ObjectMapper();
objectMapper.configure(SerializationFeature.FAIL_ON_EMPTY_BEANS, false);
objectMapper.configure(MapperFeature.CAN_OVERRIDE_ACCESS_MODIFIERS, true);

// 简化数据结构
public class LongTermMemory {
    private String userId;
    private UserPreferences preferences; // POJO替代Map
    private List<SessionRecord> history;
}

// 自定义Serializer
public class UserPreferencesSerializer extends JsonSerializer<UserPreferences> {
    @Override
    public void serialize(UserPreferences value, JsonGenerator gen, SerializerProvider serializers) 
            throws IOException {
        gen.writeObject(value.toMap()); // 转换为Map
    }
}
```

### 陷阱4：Redis缓存穿透

**问题描述**：
```
问题：Redis中不存在的数据频繁请求数据库
症状：
- 数据库压力大
- 响应时间长
- 缓存命中率低

原因：
- Redis未命中时直接查询数据库
- 未对空结果缓存
- 未设置热点数据预热
```

**解决方案**：

| 方案 | 说明 | 适用场景 |
|------|------|---------|
| **缓存空结果** | Redis未命中时缓存空对象（TTL短） | 高频查询场景 |
| **布隆过滤器** | 预检查key是否存在 | 大规模数据场景 |
| **热点数据预热** | 启动时预热常用用户数据 | 用户量少场景 |

**示例代码**：
```java
// 缓存空结果
public LongTermMemory get(String userId) {
    String cacheKey = "user:" + userId;
    String json = redisTemplate.opsForValue().get(cacheKey);
    
    if (json == null) {
        // 缓存未命中，查询数据库
        LongTermMemory memory = loadFromDatabase(userId);
        
        // 缓存空结果（TTL: 5分钟）
        if (memory == null) {
            redisTemplate.opsForValue().set(cacheKey, "{}", Duration.ofMinutes(5));
            return null;
        }
        
        // 缓存查询结果
        redisTemplate.opsForValue().set(cacheKey, memory.toJson(), Duration.ofHours(1));
        return memory;
    }
    
    return LongTermMemory.fromJson(json);
}
```

## RAG陷阱

### 陷阱5：Chroma连接超时

**问题描述**：
```
问题：Chroma客户端连接超时
症状：
- `Connection refused`异常
- `SocketTimeoutException`
- 检索失败

原因：
- Chroma服务未启动
- 网络延迟高
- 端口配置错误
```

**解决方案**：

| 方案 | 说明 | 适用场景 |
|------|------|---------|
| **健康检查** | 启动前检查Chroma服务可用性 | 开发/测试环境 |
| **重试机制** | 检索失败时自动重试（3次） | 生产环境 |
| **降级方案** | Chroma不可用时使用本地文件检索 | 容灾场景 |

**示例代码**：
```java
// 健康检查
public void healthCheck() {
    try {
        // Ping Chroma服务
        String healthUrl = chromaEndpoint + "/api/v1/heartbeat";
        HttpClient client = HttpClient.newHttpClient();
        HttpRequest request = HttpRequest.newBuilder()
            .uri(URI.create(healthUrl))
            .GET()
            .build();
        
        HttpResponse<String> response = client.send(request, HttpResponse.BodyHandlers.ofString());
        if (response.statusCode() == 200) {
            logger.info("Chroma health check passed");
        } else {
            throw new RuntimeException("Chroma health check failed: " + response.body());
        }
    } catch (Exception e) {
        throw new RuntimeException("Chroma connection failed", e);
    }
}

// 重试机制
public List<String> retrieve(String query, int k) {
    int maxRetries = 3;
    for (int i = 0; i < maxRetries; i++) {
        try {
            return client.query("documents", 
                List.of(queryEmbedding), k).getDocuments();
        } catch (Exception e) {
            logger.warn("Chroma query failed (attempt {}/{}): {}", i + 1, maxRetries, e.getMessage());
            if (i == maxRetries - 1) {
                throw new RuntimeException("Chroma query failed after " + maxRetries + " retries", e);
            }
            // 重试间隔
            try {
                Thread.sleep(1000 * (i + 1));
            } catch (InterruptedException ie) {
                Thread.currentThread().interrupt();
            }
        }
    }
    return Collections.emptyList();
}
```

### 陷阱6：向量化内存溢出

**问题描述**：
```
问题：Sentence Transformer向量化大量文本时内存溢出
症状：
- `OutOfMemoryError: Java heap space`
- JVM崩溃
- 向量化失败

原因：
- 向量化模型加载到内存
- 批量处理文本过多
- 未启用GPU加速
```

**解决方案**：

| 方案 | 说明 | 适用场景 |
|------|------|---------|
| **分批处理** | 每次处理少量文本（如100条） | 通用场景 |
| **启用GPU** | 使用CUDA加速向量化 | GPU可用场景 |
| **降级模型** | 使用更小的模型（如all-MiniLM-L6-v2） | 资源受限场景 |

**示例代码**：
```java
// 分批处理
public void batchEmbedding(List<String> texts, int batchSize) {
    for (int i = 0; i < texts.size(); i += batchSize) {
        List<String> batch = texts.subList(i, Math.min(i + batchSize, texts.size()));
        
        // 向量化批量文本
        List<Embedding> embeddings = sentenceTransformer.encode(batch);
        
        // 存储到Chroma
        storeEmbeddings(embeddings);
    }
}

// 降级模型
// 选择更小的模型（内存占用少）
SentenceTransformer model = new SentenceTransformer("all-MiniLM-L6-v2");
```

## 通用调试技巧

### 陷阱7：调试信息不足

**问题描述**：
```
问题：调试时缺乏足够信息定位问题
症状：
- 难以复现问题
- 日志信息不足
- 无法追踪调用链
```

**解决方案**：

| 方案 | 说明 | 适用场景 |
|------|------|---------|
| **增强日志** | 记录关键变量、调用栈、耗时 | 开发/测试环境 |
| **Trace ID** | 为每个请求生成唯一Trace ID | 生产环境 |
| **性能监控** | 记录API调用耗时、数据库查询耗时 | 生产环境 |

**示例代码**：
```java
// 增强日志
logger.info("ShortTermMemory.add() - message: {}, history size: {}", 
    msg.getContent(), history.size());

// Trace ID
public class RequestTrace {
    private static final ThreadLocal<String> traceId = ThreadLocal.withInitial(
        () -> UUID.randomUUID().toString().substring(0, 8)
    );
    
    public static String getTraceId() {
        return traceId.get();
    }
}

// 性能监控
public Response run(String input) {
    long startTime = System.currentTimeMillis();
    
    // ... 执行逻辑 ...
    
    long duration = System.currentTimeMillis() - startTime;
    logger.info("Agent.run() - traceId: {}, duration: {}ms", 
        RequestTrace.getTraceId(), duration);
    
    return response;
}
```
