### 11.6.2 代码改错（每题15分，共30分）

#### 1. Docker镜像优化改错

```dockerfile
# ❌ 错误代码（Dockerfile）
FROM openjdk:17
COPY target/app.jar app.jar
ENTRYPOINT ["java", "-jar", "app.jar"]
```

**问题**：  
1. 未使用多阶段构建  
2. 未精简基础镜像

**正确代码**：
```dockerfile
# ✅ 正确代码（Dockerfile）
# 第一阶段：构建
FROM maven:3.8-openjdk-17 AS builder
WORKDIR /build
COPY pom.xml .
RUN mvn dependency:go-offline -B
COPY src ./src
RUN mvn package -DskipTests

# 第二阶段：运行
FROM openjdk:17-jdk-slim
WORKDIR /app
COPY --from=builder /build/target/app.jar app.jar
EXPOSE 8080
ENTRYPOINT ["java", "-jar", "app.jar"]
```

---

#### 2. Prometheus配置改错

```yaml
# ❌ 错误代码（prometheus.yml）
global:
  scrape_interval: 15s

scrape_configs:
  - job_name: 'agent-app'
    static_configs:
      - targets: ['localhost:8080']  # ❌ Docker网络中应使用服务名
    metrics_path: /actuator/prometheus
```

**问题**：  
1. Docker网络中应使用服务名而非localhost

**正确代码**：
```yaml
# ✅ 正确代码（prometheus.yml）
global:
  scrape_interval: 15s

scrape_configs:
  - job_name: 'agent-app'
    static_configs:
      - targets: ['agent-app-service:8080']  # ✅ 使用Docker网络地址
    metrics_path: /actuator/prometheus
```

### 11.6.3 小型设计（30分）

#### 设计任务：实现告警服务

**需求**：  
设计一个`AlertService`类，实现钉钉和邮件告警。

**功能要求**：  
1. 钉钉告警（使用Webhook URL）  
2. 邮件告警（使用SMTP配置）  
3. 方法签名：`void notify(String title, String message)`

**参考答案**：
```java
@Service
public class AlertService {
    private final DingTalkNotifier dingTalkNotifier;
    private final EmailNotifier emailNotifier;
    
    public AlertService(DingTalkNotifier dingTalkNotifier, EmailNotifier emailNotifier) {
        this.dingTalkNotifier = dingTalkNotifier;
        this.emailNotifier = emailNotifier;
    }

    public void notify(String title, String message) {
        // 钉钉告警
        dingTalkNotifier.notify(title, message);
        
        // 邮件告警（可选）
        // emailNotifier.notify("admin@example.com", title, message);
    }
}
```

### 11.6.4 拓展挑战（20分）

#### 挑战1：Kubernetes部署优化

**背景**：  
当前Kubernetes部署缺少健康检查与资源限制。

**问题**：  
1. 如何添加健康检查？  
2. 如何配置资源限制？  
3. 如何实现自动扩缩容？

**参考答案**：

```yaml
# deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: agent-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: agent-app
  template:
    metadata:
      labels:
        app: agent-app
    spec:
      containers:
      - name: agent-app
        image: agent-app:latest
        ports:
        - containerPort: 8080
        resources:
          requests:
            memory: "256Mi"
            cpu: "100m"
          limits:
            memory: "512Mi"
            cpu: "500m"
        livenessProbe:
          httpGet:
            path: /actuator/health
            port: 8080
          initialDelaySeconds: 10
          periodSeconds: 5
        readinessProbe:
          httpGet:
            path: /actuator/ready
            port: 8080
          initialDelaySeconds: 5
          periodSeconds: 3
---
# horizontal-pod-autoscaler.yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: agent-app-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: agent-app
  minReplicas: 2
  maxReplicas: 10
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 80
```

**优化策略**：
- 添加健康检查（livenessProbe、readinessProbe）
- 配置资源限制（requests、limits）
- 实现自动扩缩容（HPA）

#### 挑战2：日志分析系统搭建

**背景**：  
当前系统日志分散，无法统一分析。

**问题**：  
1. 如何选择ELK或Loki方案？  
2. 如何配置日志收集与索引？  
3. 如何实现日志可视化？

**参考答案**：

```yaml
# loki/promtail-config.yml
server:
  http_listen_port: 9080
  grpc_listen_port: 0

positions:
  filename: /tmp/positions.yaml

clients:
  - url: http://loki:3100/loki/api/v1/push

scrape_configs:
  - job_name: application
    static_configs:
      - targets:
          - localhost
        labels:
          job: app
          __path__: /var/log/app/*.log
```

**ELK vs Loki对比**：
- **ELK**：功能全面，适合大规模日志分析  
- **Loki**：轻量级，与Prometheus集成，适合中小规模日志分析

**日志可视化**：
- ELK：使用Kibana  
- Loki：使用Grafana
